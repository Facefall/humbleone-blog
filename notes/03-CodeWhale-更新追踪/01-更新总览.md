## 本次变化摘要

### 1. 这次是明确的新版本发布：`0.8.62`

相对上一条公开基线 `0.8.61`，这次 changelog 新增了完整的 `0.8.62` 区块。对追踪工作来说，最重要的判断已经从“同版本继续补写”切换成“正式发版”：

- `0.8.62` 是新的发布边界
- `Unreleased` 顶部目前为空，说明当前重心回到刚发布的 `0.8.62`
- 这次可以明确按新版本整理功能、行为和兼容性变化，而不必停留在“正文扩写”层面

### 2. `0.8.62` 的主题是模型路由、子 Agent 默认策略、证据约束和交互补面

这一版不是单点小修，而是一组互相关联的控制面调整：

- Z.AI 直连默认模型切到 `GLM-5.2`
- GLM 家族补入 `GLM-5-Turbo`，并把 `explore` 类子 Agent 默认导向更便宜的 sibling
- `type: "explore"` 子 Agent 默认变成 `model_strength: "faster"`
- OpenAI Codex / GPT-5.5 的 faster route 不再伪造更便宜 sibling，而是诚实保留在 GPT-5.5 并把 reasoning effort 解析为 `low`
- 基础 prompt 与 delegate guidance 更明确鼓励 2-4 个只读 explore 子 Agent 并行做广域调查
- runtime guardrail 更强调“证据足够即可收束”，而不是无限搜索/无限委托

这些变化共同指向一个方向：CodeWhale 继续把多 Agent 编排从“会用”推进到“默认策略、证据边界、子父关系、成本路线都更可解释”。

### 3. 新增能力与修复同样值得看，不只是默认模型切换

`Added` 区块里有几项对日常使用影响很直接：

- 新的 `request_user_input` 工具把澄清问题从普通聊天消息升级成结构化 UI 交互
- transcript hyperlink 通过 out-of-band OSC 8 处理，减少终端列漂移和复制污染
- 新增 `[skills].scan_codewhale_only = true`，可以把 session-time skill discovery 限定在 CodeWhale 自己的目录
- `permissions.toml` 的 ask-only 规则被真正接入 TUI engine

`Fixed` 区块虽然看上去分散，但含金量不低：

- `config.toml` / `settings.toml` / `tui.toml` 重写时尽量保留注释和格式
- snapshot gate 会尊重 `[snapshots].enabled`
- Poppler 的 `pdftotext` 探测改为 `-v`，避免 `--version` 被误当成输入文件
- Plan-mode 确认弹窗现在能直接展示 active checklist

## 对使用者/维护者的判断

### 风险

- 默认模型和 explore 默认策略同时变化，可能直接改变成本、速度和输出风格
- 如果团队对 `explore` 子 Agent 仍按旧心智理解，可能误判为什么现在默认跑更快 sibling
- stronger guardrail 与 no-progress loop 限制会改变“无限找资料直到有答案”的旧行为预期
- 只保存最新 state 而不保存快照，后续仍不利于精确重建逐句变化

### 机会

- `request_user_input` 为需要用户澄清的 agent flow 提供了更正式的产品面
- `scan_codewhale_only` 给混合多工具环境的人一个更干净的 skill discovery 开关
- transcript hyperlink 和 provider metadata centralization 说明维护者在同时补用户体验和配置一致性
- `0.8.62` 对“证据驱动 synthesis”写得更明确，适合拿来校准你自己的 agent 工作流设计

## 建议动作

1. 如果你在用 Z.AI、GLM 系列或 OpenRouter 上的 GLM 路线，先复查默认模型与 faster sibling 是否符合预期。
2. 如果你依赖 `explore` 子 Agent 做仓库调查、版本调查、API surface 调查，重新检查默认 `model_strength: "faster"` 是否会影响准确率与成本平衡。
3. 如果你在设计自己的 agent orchestration，重点看 `request_user_input`、summary provenance、guardrail/no-progress loop 这三类变化。
4. 如果后续想精确记录“这一轮多了哪几句”，应把完整 changelog 快照加入私有 state，而不是只存 hash 与版本列表。
