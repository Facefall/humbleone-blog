## 每个版本做了什么

### `Unreleased`

新增 `Retroactive credits` 段落。它不是新功能发布，而是对已经发出的历史修复/贡献做补记，当前列出的项目包括：

- 全局 `~/.deepseek/AGENTS.md` fallback loading
- MCP 的 CRLF SSE 事件解析
- VTE / flicker 终端的 reduce-motion 默认行为
- LoongArch64 上的 `portable-pty` 0.9 升级
- `DEEPSEEK_ALLOW_INSECURE_HTTP` for LAN vLLM
- `reasoning_content` 保持英文
- `ExternalTool` abstraction layer
- ephemeral generated project context

这说明维护者在补全历史贡献归因；它对“历史版本理解”和“问题来源回溯”有价值，但不等价于新的运行时特性上线。

### `0.8.61`

当前 `0.8.61` 继续围绕**多 Agent 运行时控制面**展开，公开记录中的重点包括：

- WhaleFlow runtime foundations：worker runtime profile、跨 provider 模型注册、provider readiness、context budget、provider adapter、resource telemetry
- Per-role heterogeneous-model sub-agent routing：按角色分配模型/provider，例如 scout、synthesis、verifier 可使用不同路线
- Durable goal mode：目标跨轮跟踪、token/time accounting、完成前 verifier-as-judge gate
- Parent-visible worker interaction contract
- OpenAI-compatible `/v1/chat/completions` legacy app-server transport

同时，`Changed` 与 `Fixed` 显示这一版在把子 Agent 从“临时并发实体”往“durable worker substrate”推进，具体体现为：

- sidebar 不再只显示硬编码 `running`，而是投影真实 worker lifecycle state
- sub-agent 返回结构化 `needs-input checkpoint`，而不是直接停在那里
- shell 与 verifier 默认后台执行，阻塞等待需要显式触发
- Plan mode 明确为严格只读
- `/swarm` 依赖 durable worker substrate
- provider/model route 改为 session-local 隔离
- over-limit preflight 与 `context_length_exceeded` 的恢复机制更明确
- 同步工具可取消，不再长时间拖住 turn
- 多子 Agent 启动导致的 TUI freeze 被作为 launch blocker 修复

## 新功能和能力变化

这次虽然没有新版本号，但对能力图谱的信号很强：

1. CodeWhale 正在把多 Agent 能力从“能跑”推进到“有控制面、有生命周期、有路由隔离、有成本/进度统计”的阶段。
2. WhaleFlow 不再只是工作流概念，而是在 runtime foundations 层开始占位。
3. durable goal mode 说明任务不是只看单轮完成，而是开始具备跨轮目标管理与验收门禁。
4. provider/model route isolation 说明多 provider 共存环境下的运行时边界被提到更高优先级。

## 兼容性、迁移、破坏性变化或配置变化

从 changelog 能直接确认的兼容性/行为变化包括：

- Plan mode 现在被明确为严格只读，不应再假设它可直接执行 shell 工具
- shell 与 verifier 工作默认后台化；如果既有自动化依赖阻塞式等待，需要复查行为预期
- provider/model route state 变为 session-local；依赖跨 session 或跨 route 隐式继承的场景需要重新验证
- `/swarm` 的可用性现在受 durable worker substrate 约束
- 旧 `deepseek` 安装/更新路径解析到 `codewhale`

没有看到明确的“必须手动迁移配置文件”说明，但多处运行时语义已变化，实际使用上应视为需要回归验证的一次更新。

## 值得重点关注的风险或机会

### 风险

- 多 Agent 执行语义在收敛，旧脚本或旧心智模型可能与当前 worker lifecycle 不一致
- provider/model route isolation 可能暴露之前被共享状态掩盖的问题
- changelog 在同一版本号下持续更新，若只订阅“新版本出现”会漏掉重要变更

### 机会

- 多 Agent 稳定性显著加强，适合继续观察其是否能承担更长任务链
- WhaleFlow runtime foundations 与 goal mode 的组合，可能成为后续自动化编排和验收闭环的基础
- 对需要多模型协作的用户，per-role model routing 的方向值得重点跟踪

## 对使用者/维护者的建议动作

1. 把 `0.8.61` 视为一次“运行时语义升级”而不只是普通补丁版。
2. 如果你依赖 sub-agent、provider 切换、verifier、goal 管理或 `/swarm`，应做一次针对行为语义的回归检查。
3. 后续追踪中，不要只看版本号；同版本下 changelog 内容增补也需要进入日常情报流。
