## 每个版本做了什么

### `0.8.62`

这一版的重心非常清晰，不是单一功能，而是一组围绕模型路由、子 Agent 默认策略、交互补面、证据约束和配置一致性的系统性调整。

#### 1. Changed

- Z.AI 直连默认模型从此前的默认逻辑切到 `GLM-5.2`，文档和示例配置也不再把它表述为 opt-in preview。
- GLM 家族补入了真正的 `GLM-5-Turbo`，并把 `GLM-5.2` 父任务的 faster/explore 子任务默认导向 `GLM-5-Turbo`，而不是降回 `GLM-5.1`。
- `type: "explore"` 子 Agent 现在默认使用 `model_strength: "faster"`。这意味着“受限范围内的只读查询/搜索/状态查看”默认会走更便宜的同家族 sibling，除非显式指定 `model` 或 `model_strength: "same"`。
- GPT-5.5 / OpenAI Codex 的 faster route 不再伪造一个并不存在的更便宜同 provider sibling，而是继续停留在 GPT-5.5，并把 reasoning effort 诚实解析为 `low`。
- base prompt 和 delegate skill guidance 更明确鼓励 2-4 个只读 `explore` 子 Agent 并行做大范围仓库、版本、分支、benchmark、API surface 调查，而把架构、集成与最终验证留给父 Agent。
- runtime guardrail 进一步强调“补齐缺失事实后就应收束”，并新增更早阻止重复只读/重复委托调用、限制单回合同类无进展 broad lookup/delegation loop 的机制。
- sub-agent handoff 和可见性被继续加强：直接子 Agent 的完成结果会更快唤醒父 Agent；嵌套子 Agent 的完成通知只回到直接父级，避免主模型被孙级结果淹没；sidebar 也按父子树展示层级。
- sub-agent summary provenance 被写得更严格：子 Agent 自述结果明确被当成“未验证自报告”，并通过 `summary_kind: complete | truncated` 告诉父模型看到的是完整摘要还是截断摘要。
- provider metadata 被集中到共享 `ProviderMetadata` registry，减少 provider picker、`codewhale auth`、`doctor --json`、setup hints 之间的漂移。

#### 2. Added

- 新增 `request_user_input` 工具。Agent 现在可以通过结构化 modal UI 向用户提 1-3 个澄清问题，而不是只能在聊天里发一句希望用户看见的话。这个能力覆盖了工具层、engine、TUI modal 和 runtime protocol。
- 新增 transcript hyperlink 的 out-of-band OSC 8 路径。文件、文件行号和 URL 链接可以更稳地在终端中变成可点击链接，同时避免 `ESC` 控制序列占据显示列或污染复制内容。
- 新增 `[skills].scan_codewhale_only = true`。开启后，session-time skill discovery 只扫描 CodeWhale 自己的 skill roots，不再混扫 `.claude/skills`、`.cursor/skills`、`~/.agents/skills` 等跨工具目录。
- 新增 permission/ask runtime rules 接入。兄弟目录里的 `permissions.toml` ask-only 规则现在会被 TUI engine 加载，并在 `exec_shell` 前真正生效。
- `docs/RUNTIME_API.md` 新增 `codewhale app-server --insecure-no-auth` 的 loopback-only 测试说明，并明确警告不要与 `--mobile` + `0.0.0.0` 组合使用。

#### 3. Fixed

- TUI 空启动欢迎块改为按真实渲染宽度居中；HTTP streaming body 读取错误也更明确地区分“可重试”还是“已部分输出，不能回放重试”。
- 重写 `config.toml`、`settings.toml`、`tui.toml` 时，会尽量保留用户注释和原格式；如果 merge 失败，也会回退成普通序列化输出，而不是直接写失败。
- per-tool snapshot 现在会尊重 `[snapshots].enabled`，与 session-level gate 行为一致。
- Poppler 的 `pdftotext` 探测从 `--version` 改为 `-v`，修掉只装了 Poppler 时把 `--version` 当输入文件的检测错误。
- Plan-mode 确认弹窗现在会展示 active checklist，用户可以在确认/修改计划前直接看到当前的 `checklist_write` 细项。

#### 4. Retroactive credits

`0.8.62` 还追加了一段 retrospective credit reconciliation，补记了此前已经 shipped 但没有写进 changelog 的社区修复/报告归因。这里的重点不是“本轮新增功能”，而是官方借这个版本把历史 credit 补全了，包括：

- 全局 `~/.deepseek/AGENTS.md` fallback loading
- MCP 的 CRLF SSE 事件解析
- VTE/flicker terminals 的 reduce-motion 默认行为
- `portable-pty` 0.9 / LoongArch64
- `DEEPSEEK_ALLOW_INSECURE_HTTP`
- hidden `reasoning_content` 的语言保持
- `ExternalTool` abstraction layer
- ephemeral generated project context

## 新功能和能力变化

从使用者角度，这次最值得提炼的能力变化有五条：

1. 子 Agent 默认策略更明确了，尤其是 `explore` 默认走 `faster`。
2. 模型路由更诚实了，不再在没有真实 cheaper sibling 时伪造 fallback。
3. Agent 与用户之间的澄清交互变成了一等工具能力，而不是聊天习惯。
4. 子 Agent 的摘要 provenance 被正式制度化，父 Agent 不能再把子 Agent 自述当成已验证事实。
5. Skill discovery、permission ask rules、provider metadata 这些“配置边界层”开始被系统性收紧。

## 兼容性、迁移、破坏性变化或配置变化

当前 changelog 没有写出必须手工执行的迁移步骤，但有几类行为变化需要当成“潜在兼容性变化”来理解：

- 如果你依赖 Z.AI 直连默认模型，现在默认会落在 `GLM-5.2`。
- 如果你依赖 `explore` 子 Agent 的旧默认成本/速度行为，现在默认会倾向 `faster`。
- 如果你之前默认认为子 Agent 摘要可以直接当证据引用，现在必须区分“子 Agent 自述”和“父 Agent 已复核事实”。
- 如果你的工作区混有多工具 skills，`scan_codewhale_only` 提供了新的隔离开关，但它默认仍不是开启态。
- ask-only `permissions.toml` 规则现在真的会参与 `exec_shell` 审批路径，过去依赖“看起来存在但没接入”的心智模型需要纠正。

## 值得重点关注的风险或机会

### 风险

- 默认模型和默认 faster sibling 改动，可能让成本、延迟和输出质量曲线一起变化。
- 如果团队把 `explore` 当成“和父 Agent 一样严谨”的默认模型路径，可能低估默认 cheaper sibling 带来的信息损耗风险。
- guardrail 更积极阻断无进展循环后，一些靠“继续搜、继续委托”拖出结果的工作流可能更早被迫收束。

### 机会

- `request_user_input` 让高质量澄清流真正进入产品层，可直接提升 agent 与用户对齐质量。
- summary provenance 的强化，为做更严肃的多 Agent 审计链路提供了明确制度基础。
- `scan_codewhale_only` 对混合环境用户很有价值，尤其适合减少跨工具 skill 污染。
- provider metadata centralization 说明维护者开始认真处理“多入口配置一致性”这个长期维护问题。

## 对使用者/维护者的建议动作

1. 如果你在用 Z.AI 或 GLM 系列，立刻复查默认模型是否仍符合你的成本/效果预期。
2. 如果你大量使用 `explore` 子 Agent，检查默认 `model_strength: "faster"` 会不会让某些任务需要显式改回 `same`。
3. 如果你在设计自己的 agent runtime / orchestration，重点吸收 `request_user_input`、summary provenance、no-progress loop guard 这三类设计。
4. 如果你想让后续自动简报能精确指出“本轮多了哪几句”，应把完整 changelog 快照加入私有 state，而不是只存 hash 与版本列表。
