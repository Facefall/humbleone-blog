# Coding Agent 周洞察 · 2026-06-22

## 本周检查结论

最近 `2 周` 的官方证据比上周更具体了一步：

> coding agent 的竞争，正在从“有没有 sub-agent / parallelism”收敛到“有没有一套可治理的子代理调度面”。

关键不再只是能不能开更多 agent，而是：

- 谁来决定何时准入更多子代理
- 总预算和单 worker 预算怎么约束
- 什么时候必须让用户接管
- 子代理结果怎样回收、取消、恢复、跨会话搬运
- 并发执行和安全/权限边界是不是同一套控制面

## 两周滚动证据窗口内有哪些关键变化

### OpenAI Codex

在 `2026-06-08` 到 `2026-06-22` 的窗口内，`Codex` 官方最有信息量的是 `0.141.0`（`2026-06-18`）以及随后 `0.142.0-alpha.*` 的连续迭代。可直接确认的重点包括：

- remote executors 改成认证过的端到端加密 relay
- 跨平台 remote execution 保留 executor 原生 cwd / shell / permission path
- selected executor plugins 可以按 thread 激活 stdio MCP
- app-server client 可以列 immediate child threads，并读外部 agent import 结果与 rate-limit credits
- TUI 的用户输入提示支持 inactivity auto-resolve

这里最关键的不是“多了一个功能”，而是 `thread / child-thread / remote executor / plugin MCP / user input` 已经被放进同一张运行时控制图里。

### CodeWhale

`CodeWhale` 在 `2026-06-17` 的 `0.8.62` 和 `2026-06-19` 的 `0.8.63` 上，把“子代理怎么放权”做得非常显式：

- `request_user_input` 成为正式工具
- `type: "explore"` 子代理默认走更便宜的 sibling 路由
- runtime loop guard 更早拦截重复只读循环与无进展 delegation
- 新增 sub-agent fanout safeguards：`max_admitted`、shared `token_budget`、per-worker token enforcement
- provider-specific sub-agent fanout config 可覆盖并发、深度、超时、预算
- 单一 `agent` 工具已经暴露 status / peek / cancel，且支持 `worktree: true`

这说明 `CodeWhale` 不只是“支持 sub-agent”，而是在把 `sub-agent governor` 作为一等配置面建设。

### Reasonix

`Reasonix` 在 `2026-06-17` 的 `v1.9.1` 与 `2026-06-20` 的 `v1.10.0` 里，继续把并发与恢复链路补齐：

- smart plan-mode gate 对 bash 做只读校验，阻止写能力泄漏进 plan mode
- background job artifacts 持久化、迁移与失败隔离
- project session stranded by version downgrade 的 re-home 修复
- `list_sessions` / `read_session` 成为跨 session AI context sharing 工具
- release 中还能看到 `/plan-exec`、`parallel_tasks`、goal enforcement、AutoResearch 与 shared MCP 生命周期的持续推进

它强调的是：并发不是瞬时调度，而是要落到 `session / artifact / recovery / cross-session context` 的长期工作台能力上。

### Claude Code

按本次任务给定的 source-of-truth，我检查了 `Anthropic News` 与 `Anthropic Engineering`。在最近两周窗口内，没有看到新的、显式以 `Claude Code` 为主题的 News/Engineering 文章。

为避免把窗口内信号误判成“没有动作”，我额外核对了 `Claude Code` 官方 changelog 页面。`2026-06-19` 的 `2.1.183` 明确出现两类与本周主题强相关的更新：

- auto mode safety 进一步收紧：会阻止未被用户明确要求的破坏性 git 命令与 destroy 类基础设施命令
- 多个 subagent / resume / nested load 相关问题被修复，包括 subagent 的 WebSearch 空结果、background agent 结果返回、plan mode 在 compaction 后丢失等

因此，Claude Code 这周给出的不是“更激进的 fanout 能力”，而是 `delegation + autonomy` 必须被更细粒度安全规则约束。

## 本周最值得研究的 1 个设计点

### 研究点

`子代理调度治理` 正在成为下一层竞争面：不是谁能开子代理，而是谁能把子代理并发、预算、权限、恢复和回收编成一套可治理的 supervisor plane。

## 这个设计点说明了什么架构 / 能力 / 竞争变化

### 架构变化

行业正在从“一个大 agent 调很多工具”过渡到“root agent 监管多个 worker / child-thread / background task”。  
一旦进入这个阶段，真正决定成熟度的就不是 tool list，而是：

- 准入控制
- 预算控制
- 权限控制
- 取消与回收
- artifact 持久化
- resume / re-home / cross-session continuity

### 能力变化

“会并发”已经开始商品化；“会治理并发”才是新的门槛。

### 竞争变化

- `Codex`：更像在补 infra 级 supervisor plane，强项是 remote executor、plugin/MCP、child-thread accounting
- `CodeWhale`：最明确地把 fanout governor 做成配置与 runtime 语义
- `Reasonix`：最强调并发之后的持久化、恢复、跨 session 工作台能力
- `Claude Code`：更突出 autonomy 必须与安全策略协同，而不是默认让并发扩张

## 行业共性 vs 独特点

### 行业共性

- sub-agent / child task 不再是单纯的“能不能开”，而是有 admission / budget / lifecycle 约束
- plan/read-only 与多代理调度已经绑定，不能把 plan mode 当作单纯 UI 状态
- background job / artifact / session continuity 成为必备能力
- 自动化程度越高，用户接管点和 destructive-action policy 越细

### 仍然较独特的点

- `Codex`：远程执行、插件目录、child thread、用户输入倒计时都在同一条 infra 演化线上
- `CodeWhale`：对 fanout、per-worker budget、peek/cancel、provider-specific policy 的显式程度最高
- `Reasonix`：把 job artifact、re-home、cross-session tools 做成了长期工作流能力
- `Claude Code`：对 auto mode 的安全策略收口更明确，强调“自动批准不是放弃治理”

## 对你做 coding agent 的启发

如果你以后自己做 coding agent，不能只问：

- 要不要 sub-agent
- 要不要 parallel tasks

更该先问：

1. root agent 允许多少并发 worker 进入系统
2. 预算是按 root scope 计，还是按单 worker 计，还是两者都要
3. 何时必须 ask user，而不能继续自动放权
4. 子代理失败、超时、被取消之后，结果、artifact、状态怎么回收
5. 跨 session / re-home / resume 时，调度状态是不是还能自洽

这套问题不解决，sub-agent 数量越多，系统越像不可控的并发噪声放大器。

## 本周最该产出的 1 个行动点

### 行动点

产出一张 `子代理调度治理对照卡`，外加一个最小 supervisor state machine 草图。

### notes 侧产出物

建议新增一页结构化笔记，标题可直接用：

`子代理调度治理对照卡`

最少做成 8 列：

- orchestration 单位：thread / child-thread / sub-agent / session / background job
- 准入控制：并发上限、admission queue、max depth
- 预算控制：shared budget、per-worker budget、timeout
- 用户接管：approval / request_user_input / auto mode / destructive-action block
- 观测与回收：status / peek / cancel / result attribution
- 持久化：artifact、session continuity、re-home、resume
- 隔离：sandbox、worktree、remote executor、readonly gate
- 判断：行业共性 / 暂时独特 / 你该跟进

### repo 内最小验证方向

不要直接做完整 agent。先做一个 `30-90 分钟` 的 repo-facing 最小验证：

- 在 `output/agent-supervisor-plane/README.md` 放一张 Mermaid 状态机
- 只画最小链路：`RootTask -> AdmitWorker -> RunWorker -> AskUser -> PersistArtifact -> ResumeOrCancel -> Collect -> Complete`
- 对 4 家产品逐项标注：谁已经公开体现了这个状态转换，谁还没有

这样你的收获不是“又看了一堆 release note”，而是把竞争情报沉淀成自己能复用的运行时设计框架。
